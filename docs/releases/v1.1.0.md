# v1.1.0 — Citation Audit: Standalone + Memo/Opinion Workflow (2026-04-24)

> **Recommended:** `git pull` at your earliest convenience. No breaking changes; no manual migration required.
>
> This feature release builds on `v1.0.0 — Security Hardening & Ingest Robustness` and keeps existing configuration, knowledge, library, and output artifacts compatible.

## What changed

Adds a citation-audit capability with two invocation contexts:

1. **Standalone `/audit` command** — run on any existing markdown deliverable, independent of the 8-step research pipeline. Useful as a second-review pass on research memos, opinions, and summaries before they leave the office (including documents produced outside this agent).
2. **Workflow Step 9 (conditional)** — for memo and opinion deliverables, runs automatically after Step 8 and **folds the audit result into the final saved artifact** as a `## 부록: 검증 로그 (Citation Audit Log)` appendix. No separate file, no extra command.

### Core components

- **`citation-auditor` skill** (`.claude/skills/citation-auditor/`) — chunks a markdown file, extracts factual and citation-bearing claims (skips forecasts, opinions, rumors), routes each claim to per-jurisdiction verifier subagents, aggregates verdicts, and returns audited markdown.
- **Verifier plugin family** (`.claude/skills/verifiers/`) — seven per-jurisdiction / per-domain verifiers: `korean-law`, `us-law`, `eu-law`, `uk-law`, `scholarly`, `wikipedia`, `general-web`. Each declares authority weight and URL/text patterns for claim routing.
- **`/audit` slash command** (`.claude/commands/audit.md`) — user-triggered entry point: `/audit path/to/deliverable.md`.
- **`citation_auditor` Python package** — backing pipeline for chunking (token-aware), aggregation (authority-weighted voting with `verified` / `contradicted` / `unknown` labels), annotated-markdown rendering (two modes: `inline` and `append`), and a dedicated Korean-law module.

### Workflow integration — Step 9

Step 9 is **automatic** and fires when **any** of the following holds:
- Output mode is **B** (Comparative Matrix), **C** (Enforcement & Case Law), or **D** (Black-letter & Commentary).
- `legal-opinion-formatter` was invoked during Step 7 (`법률 의견서`, `opinion letter`, `legal opinion`, `formal opinion`, `opinion memo`).
- User explicitly requested a memo or opinion deliverable.

**Silently skipped** for Mode A (Executive Brief). Step 8 continues to run as the orchestrator's internal self-check; Step 9 adds **external verification** via verifier subagents — the two are complementary.

### Render modes

The Python package supports two markdown render modes, selected via `--mode` on the `render` subcommand:

- `inline` (default, used by standalone `/audit`) — per-claim badges embedded in the body + per-claim audit report at the end.
- `append` (used by Step 9 for `.md` output) — body preserved, only `[Unverified]` / `[Partially Unverified]` tags added to failing claims, plus a single appendix table at the end.

```bash
# Standalone audit (inline)
/audit output/reports/my-opinion.md

# Manual invocation of append mode
python3 -m citation_auditor render draft.md aggregated.json --mode=append
```

For Korean-law citation auditing, `korean-law-mcp` is strongly recommended. Without it, Korean statute and case verifier results may remain `unknown`.

### DOCX integration

Legal opinion deliverables in this repo are typically saved as `.docx`. Since the `citation_auditor` package is markdown-native, DOCX output goes through a **consumer adapter**:

- `scripts/docx_citation_appendix.py` (new) — two primitives:
  - `inject_unverified_tags(body_md, aggregated)` — tag failing claims in the body markdown string before it is embedded into the python-docx script.
  - `append_citation_audit_log(doc, aggregated)` — append the `부록: 검증 로그` heading, table, and disclaimer to an open `Document`, using this repo's CJK-safe font conventions.
- `scripts/citation_audit_artifacts.py` (new) — resolves the hand-off JSON in a deterministic order: explicit `--audit-json`, then `output/citation-audit-{session_id}.json`. Deprecated `output/citation-audit-latest.json` fallback is opt-in via `--use-latest-audit`.
- `scripts/citation_audit_backend.py` (new) — project-owned backend wrapper that validates aggregated verdict JSON, writes `output/claim-registry.json`, writes `output/citation-audit-{session_id}.metadata.json`, records verifier routing reasons, normalizes detailed statuses (`unsupported`, `source_unavailable`, `verifier_unavailable`), computes coverage metrics, and flags Korean-law MCP degradation without modifying the vendored `citation_auditor/` package.
- `claim_registry.schema.json`, `citation_audit_metadata.schema.json`, and `data/korean_law_aliases.json` (new) — sidecar schemas and local Korean-law alias metadata used by the backend wrapper.
- Wiring: when Step 9 fires for a `.docx` deliverable, Step 9 writes `output/citation-audit-{session_id}.json` and hands control back to Step 7's DOCX render script, which is invoked with `--audit-json` or `--session-id`. The DOCX helper validates claim spans before inline tag insertion; if a span cannot be matched safely, it skips inline insertion and records the issue in the appendix status block. See `docs/citation-audit.md` for the exact contract.

Vendor note: the current hardening path keeps citation-audit integration in project-owned adapters and sidecar metadata. Vendored skill/verifier files and the vendored `citation_auditor/` runtime should still be refreshed through upstream/fork/overlay policy rather than edited in place.

### Prompt and documentation slimming

- `docs/citation-audit.md` is now the canonical citation audit spec for triggers, artifacts, status labels, format support, and Korean-law MCP degradation.
- `.claude/routing/skills.yaml` centralizes skill routing and deep-researcher thresholds.
- `references/source-payload-contract.md` defines the compact retrieval envelope so downstream prompts receive sanitized relevant passages instead of full source text by default.

### Output shape (Step 9 / append mode)

A Mode B/C/D or opinion deliverable now ends with:

```markdown
...body of the memo, with [Unverified] / [Partially Unverified] tags on any failing claims...

---

## 부록: 검증 로그 (Citation Audit Log)

| # | 클레임 (Claim) | 판정 (Verdict) | Verifier | 근거 (Evidence) |
|---|---|---|---|---|
| 1 | ...           | ✓ verified     | korean-law | law.go.kr/... |
| 2 | ...           | ⚠ contradicted | us-law     | — |
```

### Workflow numbering unified

This release also unifies workflow step numbering across `CLAUDE.md`, `README.md`, and `docs/*/how-to-use.md`. Previous documentation interleaved two schemes (e.g., `3.5` in README vs. `4` in CLAUDE.md); all docs now use the same linear `0–8 + conditional 9` sequence.

### What this is not

- Not a substitute for Step 4 fact-checker. Step 4 intercepts hallucinations *before* analysis; Step 9 / `/audit` verifies citations *after* a deliverable exists.
- Not a citable primary source. Audit verdicts are guidance, not authority.
- Not a change to Steps 1–8. Those stay as-is.

## Dependencies

New Python dependency (declared in `requirements.txt`):

```
pydantic>=2.0
```

Install with `pip install -r requirements.txt` (or add the package to your existing venv).

## Migration

No manual migration is required. Existing `user-config.json`, `knowledge/`, `library/`, and generated output artifacts remain compatible.

The workflow gains a conditional Step 9: the next time you run a Mode B/C/D or opinion query, the citation audit will fire automatically. Mode A briefs are unaffected.

If you prefer no automatic audit, request Mode A or fall back to `/audit` manually after Step 8.

# v1.1.0 — 인용 감사: 스탠드얼론 + 메모/의견서 워크플로우 통합 (2026-04-24)

> **권장:** 가급적 빨리 `git pull`. 파괴적 변경(Breaking Change) 없음. 수동 마이그레이션 불필요.
>
> 이 기능 릴리즈는 `v1.0.0 — 보안 강화 및 Ingest 견고성 개선` 이후 버전이며, 기존 설정·knowledge·library·생성 산출물과 호환됩니다.

## 변경 사항

두 가지 실행 경로를 가진 인용 감사 기능을 추가합니다:

1. **스탠드얼론 `/audit` 명령** — 8단계 리서치 파이프라인과 독립적으로, 이미 존재하는 마크다운 산출물에 감사를 실행. 리서치 메모·의견서·요약본이 외부로 나가기 전 2차 검토용(이 에이전트 밖에서 작성된 문서 포함).
2. **워크플로우 Step 9 (조건부)** — memo·의견서 산출물일 때 Step 8 이후 자동 실행되어 감사 결과가 **최종 저장 파일에 합쳐져서** `## 부록: 검증 로그 (Citation Audit Log)` 형태로 들어갑니다. 별도 파일이나 추가 명령 불필요.

### 핵심 구성요소

- **`citation-auditor` 스킬** (`.claude/skills/citation-auditor/`) — 마크다운 파일을 청크로 분할하고, 사실·인용 기반 클레임만 추출(예측·의견·소문은 제외)한 뒤, 각 클레임을 관할권별 verifier 서브에이전트로 라우팅하고, 판정을 집계해 감사된 마크다운을 반환합니다.
- **Verifier 플러그인 계열** (`.claude/skills/verifiers/`) — 7개의 관할권·도메인별 verifier: `korean-law`, `us-law`, `eu-law`, `uk-law`, `scholarly`, `wikipedia`, `general-web`. 각각 권위도(authority) 가중치와 URL/텍스트 패턴을 선언해 클레임 라우팅에 사용됩니다.
- **`/audit` 슬래시 커맨드** (`.claude/commands/audit.md`) — 사용자 수동 진입점: `/audit path/to/deliverable.md`.
- **`citation_auditor` 파이썬 패키지** — 청킹(토큰 기반), 집계(권위도 가중 투표, `verified` / `contradicted` / `unknown` 라벨), 마크다운 렌더링(`inline` / `append` 두 가지 모드), 한국법 전용 모듈로 구성된 백엔드 파이프라인.

### 워크플로우 통합 — Step 9

아래 중 **하나라도** 해당하면 Step 9 자동 실행:
- 출력 모드가 **B** (Comparative Matrix), **C** (Enforcement & Case Law), **D** (Black-letter & Commentary).
- Step 7에서 `legal-opinion-formatter`가 호출됨 (`법률 의견서`, `opinion letter`, `legal opinion`, `formal opinion`, `opinion memo`).
- 사용자가 memo·의견서 산출물을 명시적으로 요청.

**Mode A는 조용히 스킵됩니다** (Executive Brief). Step 8은 그대로 자기 검증으로 실행되고, Step 9는 verifier 서브에이전트를 통한 **외부 검증**을 추가 — 두 단계는 보완 관계입니다.

### 렌더 모드

파이썬 패키지의 `render` 서브커맨드에 `--mode` 플래그 지원 (마크다운 전용):

- `inline` (기본, 스탠드얼론 `/audit` 사용) — 본문에 클레임별 배지 + 클레임별 상세 보고서.
- `append` (Step 9의 `.md` 출력 경로에서 사용) — 본문은 그대로 두되 실패 클레임에만 `[Unverified]` / `[Partially Unverified]` 태그 추가, 말미에 부록 테이블 1개.

```bash
# 스탠드얼론 감사 (인라인)
/audit output/reports/my-opinion.md

# append 모드 수동 실행
python3 -m citation_auditor render draft.md aggregated.json --mode=append
```

한국법 인용 감사에는 `korean-law-mcp` 사용을 강력 권장합니다. 없으면 한국 법령·판례 verifier 결과가 `unknown`으로 남을 수 있습니다.

### DOCX 통합

이 레포의 법률 의견서는 주로 `.docx`로 저장됩니다. `citation_auditor` 패키지는 markdown 전용이므로, DOCX 출력은 **소비자 어댑터**를 경유해 처리합니다:

- `scripts/docx_citation_appendix.py` (신규) — 두 개의 primitive:
  - `inject_unverified_tags(body_md, aggregated)` — DOCX 스크립트가 본문 markdown 문자열을 embed하기 전에 실패 클레임에 태그 주입.
  - `append_citation_audit_log(doc, aggregated)` — 이 레포의 CJK 폰트 규칙을 그대로 적용하여 열린 `Document`에 `부록: 검증 로그` heading, table, 디스클레이머를 추가.
- `scripts/citation_audit_artifacts.py` (신규) — hand-off JSON을 명시 `--audit-json`, `output/citation-audit-{session_id}.json` 순서로 결정합니다. Deprecated `output/citation-audit-latest.json` fallback은 `--use-latest-audit`를 명시한 경우에만 사용합니다.
- `scripts/citation_audit_backend.py` (신규) — vendored `citation_auditor/` 패키지를 직접 수정하지 않고 aggregated verdict JSON을 검증하고, `output/claim-registry.json`, `output/citation-audit-{session_id}.metadata.json`, verifier routing 이유, 상세 상태(`unsupported`, `source_unavailable`, `verifier_unavailable`), coverage metrics, Korean-law MCP degradation을 기록하는 project-owned backend wrapper.
- `claim_registry.schema.json`, `citation_audit_metadata.schema.json`, `data/korean_law_aliases.json` (신규) — backend wrapper가 사용하는 sidecar schema와 로컬 한국법 alias metadata.
- 배선: `.docx` 산출물에서 Step 9가 실행되면, Step 9는 `output/citation-audit-{session_id}.json`을 쓰고 Step 7의 DOCX 렌더 스크립트에 `--audit-json` 또는 `--session-id`로 제어권을 넘깁니다. DOCX 헬퍼는 inline tag 삽입 전에 claim span을 검증하며, 안전하게 매칭되지 않으면 본문 삽입을 건너뛰고 appendix status block에 기록합니다. 상세 절차는 `docs/citation-audit.md` 참고.

Vendor 참고: 현재 hardening 경로는 citation-audit 통합을 project-owned adapter와 sidecar metadata에 둡니다. Vendored skill/verifier 파일과 vendored `citation_auditor/` runtime은 직접 수정하지 말고 upstream/fork/overlay 정책을 통해 갱신해야 합니다.

### Prompt 및 문서 경량화

- `docs/citation-audit.md`가 trigger, artifact, status label, format support, Korean-law MCP degradation에 관한 canonical citation audit spec이 되었습니다.
- `.claude/routing/skills.yaml`에 skill routing과 deep-researcher threshold를 중앙화했습니다.
- `references/source-payload-contract.md`에 compact retrieval envelope을 정의하여 downstream prompt에는 원문 전문 대신 sanitizer를 통과한 relevant passage 중심으로 전달합니다.

### 산출물 모양 (Step 9 / append 모드)

Mode B/C/D 또는 의견서 산출물은 이제 아래처럼 끝납니다:

```markdown
...메모 본문, 실패 클레임에만 [Unverified] / [Partially Unverified] 태그...

---

## 부록: 검증 로그 (Citation Audit Log)

| # | 클레임 (Claim) | 판정 (Verdict) | Verifier | 근거 (Evidence) |
|---|---|---|---|---|
| 1 | ...           | ✓ verified     | korean-law | law.go.kr/... |
| 2 | ...           | ⚠ contradicted | us-law     | — |
```

### 워크플로우 번호 체계 통일

이번 릴리즈에서 `CLAUDE.md`, `README.md`, `docs/*/how-to-use.md`의 워크플로우 단계 번호를 일원화했습니다. 기존에는 두 체계가 혼재되어 있었으나(예: README의 `3.5` vs. CLAUDE.md의 `4`), 이제 모든 문서가 동일한 `0–8 + 조건부 9` 선형 순서를 사용합니다.

### 이 기능이 아닌 것

- Step 4 fact-checker의 대체재가 아닙니다. Step 4는 분석 *이전에* 할루시네이션을 차단하고, Step 9 / `/audit`는 산출물 생성 *이후에* 인용을 검증합니다.
- 인용 가능한 1차 자료가 아닙니다. 감사 결과는 참고 자료이지 법적 권위가 아닙니다.
- Steps 1–8 변경 없음.

## 의존성

새로운 파이썬 의존성(`requirements.txt`에 선언):

```
pydantic>=2.0
```

`pip install -r requirements.txt`로 설치하거나 기존 venv에 해당 패키지를 추가하시면 됩니다.

## 마이그레이션

수동 마이그레이션은 불필요합니다. 기존 `user-config.json`, `knowledge/`, `library/`, 기존 생성 산출물은 그대로 호환됩니다.

다만 워크플로우에는 조건부 Step 9가 추가됩니다. 다음 번 Mode B/C/D 또는 의견서 쿼리 실행 시 citation audit이 자동으로 동작하며, Mode A 브리프는 영향받지 않습니다.

자동 감사를 원하지 않으면 Mode A를 요청하거나 Step 8 이후 `/audit`를 수동으로 실행하면 됩니다.
